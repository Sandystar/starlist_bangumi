# Starlist Bangumi Design Questions

Date: 2026-05-17

This document captures the accepted answers from the grill-me design session. It
is the source of truth for the MVP behavior until superseded by a later design
document.

## Core Shape

Starlist Bangumi is a local, single-user desktop WebView tool. It scans an
OpenList inbox, uses LLM + TMDB to produce a dry-run organization plan, lets the
user review that plan, then copies validated files into media-library targets
and archives the full original source folder.

The main domain object is a `WorkPlan`: one top-level source folder represents
one work collection. A work collection may include one TV season, multiple TV
seasons, Season 0 entries, movies, or TV plus movies. The plan can therefore
produce multiple library targets while keeping exactly one archive target.

## Decision Ledger

| ID | Design Question | Accepted Decision |
| --- | --- | --- |
| Q01 | What is `source_path`? | It is a dedicated inbox, for example `/NetDisk/PikPak/Download`. Only first-level folders are scanned. |
| Q02 | Should scan filter empty or incomplete folders? | No filtering. List all first-level folders and let the user decide. |
| Q03 | What is the analysis boundary? | One top-level folder is one work collection. It is never auto-split into multiple works. |
| Q04 | Can one analysis result have multiple library targets? | Yes. A `WorkPlan` has one archive target and multiple `library_targets`. |
| Q05 | Does the TV template create a work root or season folder? | TV template creates the work root. Season directories and filenames are generated by the plan. |
| Q06 | What season naming standard is used? | Follow TMDB/Emby style: `Season 0`, `Season 1`, `Season 2`, etc. Season 0 is for specials/OAD/OVA-like entries that TMDB includes. |
| Q07 | Must TV files match a TMDB episode? | Yes. TV media-library files must match a concrete TMDB season and episode. |
| Q08 | Is LLM a suggester or mapper? | LLM decides mappings; the program validates those decisions against TMDB and safety rules. |
| Q09 | Who selects among TMDB candidates? | LLM selects from TV/Movie candidates. If uncertain, result becomes `needs_user_choice`. |
| Q10 | How are TV plus movie folders handled? | Search both TMDB TV and Movie. LLM can choose one TV series and multiple movies. |
| Q11 | Must movie files match a TMDB movie? | Yes. Movie media-library files require a concrete `tmdb_movie_id`. |
| Q12 | How does archive work? | Archive copies the full source folder by default. Source deletion is explicit and happens only after successful completion. |
| Q13 | Does media-library organization move files? | No. Media-library organization only copies. It never moves, cuts, or mutates source files. |
| Q14 | How are destructive pre-clean options represented? | Split into `delete same-name mapped files before organize` and `delete run-specific archive target before archive copy`. |
| Q15 | What if media target exists without clear option? | Task fails and logs conflicting paths. |
| Q16 | What if archive target exists without overwrite option? | Task fails and asks the user to enable archive target deletion or change the template. |
| Q17 | What is organize pipeline order? | `preflight -> archive_copy -> library_prepare -> library_copy -> library_rename -> verify -> cleanup_source -> complete`. |
| Q18 | Is automatic organize allowed? | No. User must explicitly review dry-run and trigger organize. |
| Q19 | What is `needs_review`? | A valid-ish plan with information gaps: candidate uncertainty, rejected mappings, missing components, missing expected episodes, or ambiguous completeness. |
| Q20 | How are missing TMDB episodes shown? | As a separate `missing_tmdb_episodes` block, not mixed with rejected mappings. |
| Q21 | How is missing-episode scope determined? | TMDB fetched episode details are the source of truth. Every fetched TMDB season/episode must map to a source video; extra source videos may remain unmapped. |
| Q22 | How is movie coverage represented? | Movies use `selected_movie_ids`, `mapped_movies`, `missing_movies`, and unmapped movie-like files rather than complete/partial. |
| Q23 | How are subtitles handled? | External subtitles only enter the media library when bound to a validated video mapping. |
| Q24 | Which file types enter media library? | Only video files and bound external subtitles. Everything else only goes to archive. |
| Q25 | How are subtitle filenames generated? | Replace the main stem with the target video stem, preserving language/type suffixes such as `.sc.ass` or `.zh-Hans.ass`. |
| Q26 | How strict is LLM mapping JSON? | Strict fixed schema. Parse/schema failure makes the analysis task fail. |
| Q27 | How many LLM calls are used? | Three: `identify_work`, `select_candidates`, `decide_mappings`. |
| Q28 | Which TMDB season details are fetched? | The program fetches every TMDB TV season with `episode_count > 0` from selected TV details. LLM `season_hints` and `season_numbers_to_fetch` are not the source of truth. |
| Q29 | What happens after manual candidate selection? | Re-run from `decide_mappings`, not from identification/search. |
| Q30 | Are tasks/results persistent? | CLI-first direction persists each run as files under a per-run folder. GUI task persistence is no longer the core path. |
| Q31 | What storage is used? | `data/config.json` for config; `data/runs/{timestamp}-{source}/` for run records. SQLite is deprecated with the GUI path. |
| Q32 | Are LLM requests/responses saved? | Yes. CLI runs save request/response JSON and readable prompt text files in the run folder, excluding API keys and passwords. |
| Q33 | How are logs cleaned? | MVP keeps all history and provides manual clear actions only. |
| Q34 | Are tasks concurrent? | MVP is fully serial. |
| Q35 | One queue or separate queues? | One global serial queue. |
| Q36 | Can tasks be cancelled? | Queued tasks can be cancelled. Running tasks cannot be force-cancelled in MVP. |
| Q37 | How does organize retry work? | Reuse the original dry-run plan. Do not reanalyze. |
| Q38 | How does analysis retry work? | Reuse source path, prompt, and any manual candidate selection. Changed prompt creates a new analysis task. |
| Q39 | Can the same source be submitted twice? | Skip duplicate queued/running analysis for the same source. Completed sources can be analyzed again. |
| Q40 | Are analysis results versioned? | Yes, per source path. Organize tasks bind to a specific analysis version. |
| Q41 | Which results show by default? | Latest analysis result per source path. Historical versions are hidden behind a toggle. |
| Q42 | What happens to scan items after organize? | If source is not deleted, mark as organized and do not select by default. If deleted, it disappears on next scan. |
| Q43 | When do config changes take effect? | Config is read at startup. Saving during runtime writes `data/config.json` but only takes effect after restart. |
| Q44 | What config does OpenList test use? | Current form values, without affecting runtime config. |
| Q45 | What config does LLM model listing use? | Current form values, without affecting runtime config. |
| Q46 | What save-settings message is shown? | `已保存到 data/config.json。当前运行中的任务仍使用启动时配置；重启程序后新配置生效。` |
| Q47 | Are secrets plaintext? | MVP stores local single-user secrets in plaintext config. Document this clearly. |
| Q48 | Are multiple profiles supported? | No. MVP uses one config profile. |
| Q49 | Is Bangumi.tv used? | No. MVP uses TMDB only. Bangumi.tv may be future auxiliary metadata. |
| Q50 | How is TMDB language used? | Used for search/details. Names prefer current language, then original title, then LLM title. |
| Q51 | Are episode titles included in filenames? | Default yes, with config `include_episode_title_in_filename`. If missing/disabled, omit episode title. |
| Q52 | Do movie paths include year? | Yes. Movie folder and file both include year. |
| Q53 | What is default TV root template? | `{year}/{title} ({year}) [tmdbid={tmdb_id}]`. `{tvdb_id}` remains supported. |
| Q54 | What does archive template use? | Default `{year}/{title}`. Also support `{source_name}`. |
| Q55 | Are conflict suffixes auto-added? | No. Conflicts fail unless clear/overwrite options are enabled or template changes. |
| Q56 | What template variables exist? | `{title}`, `{original_title}`, `{year}`, `{tmdb_id}`, `{tvdb_id}`, `{source_name}`, `{media_type}`. |
| Q57 | How do OpenList retries work? | Network/timeout/5xx retry; 4xx parameter errors fail. `operation_interval_seconds` applies between any two OpenList operations. |
| Q58 | Where is OpenList refresh used? | Inbox scan uses config. Analysis tree root may refresh. Subdirectories and preflight default false. |
| Q59 | What are file-tree limits? | Defaults: depth 6, nodes 1000. Exposed in settings. |
| Q60 | What is sent to mapping LLM? | Compact selected TMDB details plus the grouped candidate file list: all videos plus subtitles that cannot be filename-matched to a video. The full text tree is kept as a debug artifact but is not sent to `decide_mappings`. |
| Q61 | Can LLM reference arbitrary source files? | No. Source paths must come from candidate media-file list. |
| Q62 | Candidate extensions? | Video: `.mkv .mp4 .avi .mov .m4v .ts .wmv .flv .webm`; subtitles: `.ass .ssa .srt .vtt .sub`. Configurable. |
| Q63 | Who binds subtitles? | The program binds filename-matched subtitles after a video mapping is validated. Unmatched subtitles may be sent to LLM as context, but LLM should not use them as primary media decisions. |
| Q64 | Subtitle binding rule? | Same stem or `video_stem.` prefix. Same directory first, then sibling `Subs/Subtitles`; no fuzzy matching. |
| Q65 | What is failed vs review? | External retry exhaustion or schema failure is `failed`; uncertain candidates, rejected mappings, missing selected movies, and missing TMDB episodes are `needs_review`. Extra unmapped source videos alone are allowed. |
| Q66 | Continue mapping without TMDB candidates? | No. Stop at `needs_review` and ask for prompt change or manual TMDB ID. |
| Q67 | Manual TMDB ID remapping? | MVP supports TV ID, movie IDs, and season numbers, then reruns mapping. |
| Q68 | Candidate UI style? | Show candidate cards when available and keep manual ID inputs as escape hatch. |
| Q69 | Where does result detail live? | In a right-side drawer, not in huge result cards or a separate page. |
| Q70 | Resource page layout? | Three-column workbench: scan list, analysis result list, detail drawer. |
| Q71 | Pipeline page layout? | Global queue timeline by default, with later source-path filtering. |
| Q72 | Log granularity? | Default logs are concise stage/file summaries. Detailed retry/API/LLM diagnostics are folded. |
| Q73 | Progress weights? | Analysis: identify 15, search 20, select 20, fetch 15, decide 20, validate 10. Organize: preflight 10, archive 25, prepare 10, copy 30, rename 10, verify 10, cleanup 5. |
| Q74 | Settings layout? | Left grouped navigation plus right form. |
| Q75 | Extension editor? | MVP uses comma-separated inputs, normalized on save. |
| Q76 | Where are cleanup actions? | Settings -> Tasks and Diagnostics, with irreversible confirmation. |
| Q77 | Tray/background mode? | No. Closing the window exits MVP. |
| Q78 | System notifications? | No. UI-only status. |
| Q79 | Automatic scan/analyze? | No. MVP is fully manually triggered. |
| Q80 | App login/permissions? | No. Local single-user app bound to `127.0.0.1`. |
| Q81 | Export dry-run report? | Yes. Export Markdown reports to `data/reports/`. |
| Q82 | Report filename? | `{timestamp}_{safe_title}_v{analysis_version}.md`. |
| Q83 | Prompt templates editable in UI? | No. Store prompt templates as files under `src/starlist_bangumi/prompts/`. |
| Q84 | LLM schema version? | Yes. LLM output includes `schema_version: "1.0"`. |
| Q85 | Plan schema version? | Yes. Persisted plan includes `plan_version: "1.0"`. |
| Q86 | Implementation order? | CLI analysis flow, LLM/TMDB validation, file-based run records, organize executor, verification. GUI/SQLite work is deferred. |
| Q87 | Next slice scope? | Make the CLI produce complete analysis status, validated plan, and target-side organized tree before adding copy operations. |

## WorkPlan Concept

```text
WorkPlan
  plan_version
  work_title
  source_name
  source_path
  archive_target_path
  coverage_scope
  selected_tv_series
  selected_movies
  library_targets[]
  validated_mappings[]
  rejected_mappings[]
  missing_tmdb_episodes[]
  missing_movies[]
  unmapped_files[]
```

### Library Targets

TV library targets point at a work root generated by the TV template. Season
directories are generated inside the plan:

```text
TV Root/
  Season 0/
    Title - S00E01 - Episode Name.mkv
  Season 1/
    Title - S01E01 - Episode Name.mkv
```

Movie library targets point at movie folders generated by the Movie template:

```text
Movie Title (2024) [tmdbid=123]/
  Movie Title (2024).mkv
```

## Analysis Flow

1. `identify_work`
   - Input: folder name and user prompt.
   - Output: canonical title, aliases, expected components, season hints.

2. TMDB search
   - Search both TV and Movie candidates.

3. Source tree scan
   - Scan the OpenList source folder.
   - Build a grouped video-only candidate list for mapping.

4. `select_candidates`
   - Input: folder context plus TMDB candidate pool.
   - Output: selected TV series, selected movies, or `needs_user_choice`.

5. TMDB details fetch
   - Fetch TV details, all TMDB seasons with episodes, and selected movie details.

6. `decide_mappings`
   - Input: selected TMDB data, grouped candidate files, and user prompt.
   - Output: strict schema LLM decisions.

7. Program validation
   - Ensure paths exist in candidate list.
   - Ensure TV mappings match concrete TMDB episodes.
   - Ensure Movie mappings match concrete TMDB movie IDs.
   - Bind subtitles to validated videos.
   - Produce `validated`, `rejected`, `missing`, and `unmapped` outputs.

## Execution Safety

- Media-library operations are copy-only.
- Source folder is never modified during media-library organization.
- Archive copies the full source folder.
- Source deletion can only happen after the whole pipeline succeeds and only when explicitly enabled.
- Existing media targets fail preflight unless media clear option is enabled.
- Existing archive targets fail preflight unless archive overwrite option is enabled.
- No automatic conflict suffixes are generated.

## UI Direction

Resource organization becomes a three-column workbench:

```text
Scan list | Analysis result list | Detail drawer
```

The detail drawer holds the full dry-run:

- Summary and TMDB selections.
- Library targets.
- Validated mappings.
- Missing TMDB episodes/movies.
- Rejected mappings.
- Unmapped files.
- Candidate selection / manual TMDB ID remapping.
- Organize options and submit controls.

Pipeline page is a global queue timeline. Settings page uses grouped navigation
with sections for OpenList, LLM, TMDB, Media Library Paths, Scan/Candidates,
Tasks/Diagnostics, and UI.

## Current Implementation State

The project has pivoted to a CLI-first core. The WebView UI and SQLite-backed
task history are kept as secondary/deferred surfaces, while the primary business
workflow is implemented through file-based run folders.

Implemented:

1. CLI analysis with `tools/llm_debug_pipeline.py`.
2. Three-stage LLM flow: `identify_work`, `select_candidates`, `decide_mappings`.
3. TMDB-driven validation, with all fetched TV episodes required to map to a
   source video.
4. Grouped candidate prompts with videos plus unmatched subtitles, and
   program-side same-name subtitle binding.
5. File-based run records under `data/runs/`.
6. CLI organize execution with `tools/organize_run.py`.
7. OpenList copy/archive execution with explicit destructive options.
8. Status recalculation before organize, so stale `analysis_result.json` files
   cannot bypass current validation rules.

Next implementation targets:

1. Manual TMDB remapping against an existing run folder.
2. Markdown report export from `analysis_result.json`.
3. Optional cleanup commands for old run folders and diagnostics.
4. A later UI, if needed, should consume the same file-based analysis and
   organize contracts instead of introducing a second source of truth.
